## 🎯 Project Overview

**TalentScout Hiring Assistant** is a conversational AI screening tool that automates the initial candidate-intake process for a technology-recruitment agency. It collects standard application details, lets the candidate declare their tech stack, and then dynamically generates 3–5 technical questions calibrated to that stack and the candidate's experience level — all through a clean Streamlit chat interface.

The goal is to demonstrate practical command of:

- **Prompt engineering** for both information gathering and dynamic question generation
- **LLM integration** via Groq's free-tier OpenAI-compatible API
- **Conversation state management** using a deterministic finite-state machine
- **Privacy-aware data handling** (PII hashing, anonymized storage)
- **Production-quality Python and UI design**

## 🚀 Installation

### Prerequisites

- **Python 3.10+**
- A **free Groq API key** — sign up at [console.groq.com/keys](https://console.groq.com/keys)

### Setup

```bash
# 1. Clone or unzip the repo
cd talentscout-chatbot

# 2. (Recommended) Create a virtual environment
python -m venv venv
source venv/bin/activate          # on Windows: venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Configure your Groq API key (choose ONE option)

# Option A — Streamlit secrets (recommended):
cp .streamlit/secrets.toml.example .streamlit/secrets.toml
# then edit .streamlit/secrets.toml and paste your key

# Option B — Environment variable:
cp .env.example .env
# edit .env and paste your key

# Option C — Just run the app and paste the key into the sidebar each time
```

### Run

```bash
streamlit run app.py
```

The app will open at [http://localhost:8501](http://localhost:8501).

## 📖 Usage Guide

1. **Open the app** in your browser. You'll see TalentBot's greeting.
2. **Provide your API key** in the sidebar (if not already in `secrets.toml`).
3. **Choose a model** (default: `llama-3.3-70b-versatile`).
4. **Answer the prompts** as they come — name, email, phone, experience, role, location.
5. **Declare your tech stack**, comma-separated (e.g., `Python, Django, PostgreSQL, AWS`).
6. **Answer 3–5 technical questions** generated specifically for your stack.
7. **Wrap up** — add any final notes (or just say "no") and you're done!
8. **Reference ID**: jot down the `TS-XXXXXX` ID shown at the end for any follow-up.

**To exit early at any time**, type one of: `bye`, `exit`, `quit`, `stop`, `goodbye`, `cancel`.

**To start over**, click **"🔄 Start New Session"** in the sidebar.

## 🔧 Technical Details

### Libraries

| Library                   | Purpose                                       | Why this one                                                                       |
| ------------------------- | --------------------------------------------- | ---------------------------------------------------------------------------------- |
| **streamlit** (≥ 1.30)    | UI framework                                  | Native chat primitives (`st.chat_message`, `st.chat_input`); zero-boilerplate      |
| **groq** (≥ 0.11)         | Official Groq Python SDK                      | OpenAI-compatible API; <1s latency on the free tier                                |
| **python-dotenv**         | `.env` loading                                | Convenience for local dev                                                          |

> Standard library used: `re`, `json`, `hashlib`, `secrets`, `dataclasses`, `enum`, `pathlib`, `threading`, `datetime`.

### Model Selection

The app exposes three Groq free-tier models in the sidebar:

| Model                          | Best for                                    |
| ------------------------------ | ------------------------------------------- |
| `llama-3.3-70b-versatile`      | **Default** — best instruction-following    |
| `llama-3.1-8b-instant`         | Lowest latency; good for cost-sensitive use |
| `openai/gpt-oss-20b`           | An open-weights middle-ground option        |

All three handle this workload comfortably. The 70B model is preferred for technical-question generation because of its better calibration to experience level.

### Architectural Decisions

1. **Hybrid orchestration (FSM + LLM).** A pure LLM-driven dialogue is unreliable for collecting structured fields — the model can skip steps, hallucinate values, or be derailed by adversarial input. Instead, this app uses a **finite-state machine** for the screening flow and only calls the LLM for tasks where it adds genuine value: **(a)** generating tech questions, and **(b)** off-topic fallback redirects. This keeps the app fast, cheap, predictable, and auditable.

2. **Deterministic validation.** Email, phone, name, and experience are validated with regex/word-mapping rules — no LLM involvement. This is faster (~0 ms vs ~500 ms), free, and unambiguous.

3. **No LLM access to PII.** Email and phone are validated and stored locally; they're never sent to Groq. Only the tech stack, experience, and target role are passed to the question-generation prompt.

4. **JSON-mode output with defensive parsing.** Free-tier models occasionally violate "respond with only JSON" instructions. `GroqClient._parse_json_array` strips code fences, extracts the first `[...]` substring, and falls back to numbered-list parsing — so the app degrades gracefully rather than crashing.

## 🎨 Prompt Design

Three prompt templates live in `utils/prompts.py`:

### 1. `SYSTEM_PROMPT` — Role & guardrails

This sets TalentBot's persona, scopes the assistant tightly to recruitment screening, and forbids:

- Off-topic conversation (jokes, code help, opinions)
- Hiring decisions or salary commitments
- Asking for sensitive data beyond what's required
- Revealing the system prompt itself

**Why these guardrails?** A naive recruitment chatbot is a perfect target for prompt injection — candidates could try to get it to leak the system prompt, give them the questions in advance, or write their cover letter. Tight scoping is the first line of defense.

### 2. `TECH_QUESTION_GENERATION_PROMPT` — Dynamic question generation

The heart of the project. Key design choices:

- **Difficulty calibration via experience:** explicitly instructs the model to match question depth to years of experience (0–1 → fundamentals, 2–4 → applied, 5+ → architecture/trade-offs).
- **Spread requirement:** "Cover a SPREAD of the listed technologies — don't ask 3 questions about one library." Without this, the model tends to focus on whichever tech it knows best (usually Python).
- **Format constraints:** answerable in 2–5 sentences, mix of conceptual and applied, no yes/no, no version-specific gotchas.
- **JSON-only output:** "Return ONLY a valid JSON array of strings. No markdown, no commentary, no code fences."
- **Few-shot example** of the expected JSON format.
- **Low temperature (0.6)** for consistency, but not zero — we want some variety across runs.

### 3. `FALLBACK_PROMPT` — Off-topic redirect

When a user's message doesn't fit the current stage (e.g., they ask a coding question mid-screening), this prompt instructs the model to acknowledge briefly and steer back to the expected input — without answering the off-topic question.

## 🔒 Data Privacy

Following GDPR's data-minimization principle:

| Field          | At rest in `data/candidates.jsonl`                       |
| -------------- | -------------------------------------------------------- |
| Full name      | Plaintext (needed for recruiter review)                  |
| **Email**      | **SHA-256 hash + domain only** (e.g., `gmail.com`)       |
| **Phone**      | **SHA-256 hash + last 2 digits** (for callback verify)   |
| Experience     | Plaintext                                                |
| Position       | Plaintext                                                |
| Location       | Plaintext (city/country granularity)                     |
| Tech stack     | Plaintext                                                |
| Q&A pairs      | Plaintext (the whole point of the screening)             |
| Reference ID   | Random hex token (`TS-XXXXXX`)                           |

**Additional safeguards in this demo:**

- The `.env` and `secrets.toml` files are **gitignored** — keys never hit the repo.
- The `data/candidates.jsonl` file is **gitignored** — no candidate data ever gets pushed.
- The Groq API call **never sees** candidate email/phone — only the tech stack, role, and experience.
- A thread lock around file writes makes appends atomic.

**For a production deployment, you'd additionally want:**

- Symmetric encryption (e.g., AES-GCM) so authorized recruiters can read PII back, while data-at-rest is still protected
- A "right to be forgotten" deletion endpoint
- Explicit consent capture before any data is stored
- Access logs and audit trails
- TLS-only network transport (Streamlit Cloud / a reverse proxy gives you this)

## 🧗 Challenges & Solutions

| Challenge                                                                    | Solution                                                                                                                                            |
| ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **LLM occasionally returns JSON wrapped in code fences or with preamble**    | Defensive parsing: strip ` ``` ` fences, slice from first `[` to last `]`, fall back to numbered-list extraction if JSON parse fails                |
| **Free-tier models can hallucinate field values during collection**          | Bypass the LLM for field collection entirely — use deterministic regex validators for email/phone/name/experience                                   |
| **Candidates may try to skip steps or send adversarial input**               | Finite-state machine enforces stage order; tight `SYSTEM_PROMPT` resists prompt-injection attempts                                                  |
| **Exit keywords could match inside legitimate sentences** (e.g. "I want to *exit* my current role") | Match only on whole-word boundaries with short messages (≤ 4 words); ignore substring matches in longer messages                                    |
| **Tech-stack questions tend to cluster on the model's favorite tech**        | Explicitly instructed in the prompt to "cover a SPREAD" of listed technologies                                                                      |
| **Difficulty wasn't matching experience level**                              | Added explicit calibration table to the prompt (0-1 yr / 2-4 yr / 5+ yr)                                                                            |
| **Sensitive data (email, phone) shouldn't leak to a third-party API**        | Validated and stored locally — never included in any LLM prompt                                                                                     |
| **Streamlit's session state resets on rerun**                                | All state lives in `st.session_state`; `ConversationManager` is instantiated once per session and mutated in place                                  |

## 🔮 Future Enhancements

The assignment lists these as bonus — sketches of how I'd add them:

- **Sentiment analysis:** classify each candidate response with a small Groq call after each turn; surface to the recruiter as a "confidence" signal.
- **Multilingual support:** detect the candidate's input language with `langdetect`, prepend a `Respond in {language}` instruction to the system prompt; auto-translate the greeting and stage prompts via the LLM.
- **Personalized follow-ups:** after the technical Q&A, do a single LLM call to generate 1–2 follow-up questions targeting weak spots in the candidate's answers.
- **Cloud deployment:** push to Streamlit Community Cloud (free) — `secrets.toml` is set in the dashboard; no Dockerfile needed.
- **Resume parsing:** add an `st.file_uploader` for PDF resumes; pre-fill the candidate fields with extracted text.
- **Recruiter dashboard:** a separate Streamlit page (`pages/dashboard.py`) that reads `data/candidates.jsonl` and shows screened candidates, filterable by tech stack and experience.
